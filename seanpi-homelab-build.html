I Built a Real Security Lab on a $100 Raspberry Pi

I'm studying for my ISC2 CC and Security+ certifications, and somewhere along the way I got tired of just reading about tools and decided to actually run them. This is a writeup of everything I set up on a Raspberry Pi 5 sitting on my desk — from scratch, in one session.

The Hardware

The whole thing runs on a Raspberry Pi 5 (4GB). It's a tiny computer about the size of a credit card that sips maybe 5 watts of power, which means I can leave it on 24/7 without thinking about it. I gave it a static IP at 192.168.0.2 so nothing ever moves around on me, and I access it over SSH from my Windows machine. No monitor needed.

Pi-hole + Unbound

The first real service I set up was Pi-hole, which acts as a DNS server for my whole network. Every device at home now routes its DNS requests through the Pi first. Pi-hole checks each request against a blocklist and drops anything that matches — ads, trackers, malware domains. I'm sitting at just over 1 million blocked domains across three lists (Hagezi Pro, OISD Big, Hagezi TIF).

Pi-hole alone still sends your DNS queries out to Google or Cloudflare. Unbound fixes that by resolving domains directly with the authoritative root servers — no middleman. Set it to run on 127.0.0.1:5335 and pointed Pi-hole at it. Now nobody upstream sees what I'm looking up.

Pi-hole is the bouncer. Unbound means you don't have to whisper your guest list to Google first.

Tailscale VPN

Tailscale creates an encrypted mesh network between all my devices using WireGuard under the hood. My Pi, my Windows desktop, and my iPhone are all connected with fixed Tailscale IPs regardless of where I am. I also set the Pi as an exit node so I can route my traffic through home when I'm on public WiFi. Key expiry disabled so it doesn't randomly stop working.

Security Hardening

Before exposing anything, I locked down the Pi the same way you'd harden a production Linux box. Fail2ban watches for failed SSH logins and bans the IP after too many attempts. UFW (firewall) only allows traffic on ports I actually use. SSH is configured to block root login, limit auth attempts, and timeout idle sessions. Unattended upgrades handle security patches automatically overnight.

CrowdSec

CrowdSec is an intrusion detection system with a community layer on top. It watches your logs for attack patterns — brute force, port scans, web crawlers, exploit attempts — and bans the offending IPs at the firewall level via a bouncer that hooks directly into nftables. What makes it different is the community blocklist: millions of CrowdSec users anonymously share attack data, so the moment an IP attacks someone else in the community, it gets blocked for everyone.

Here's what it blocked within the first few hours of being live:

Threat Type Blocked
HTTP crawlers 6,579
HTTP scanners 5,362
SSH brute force 2,425
HTTP brute force 496

HTTP DoS 12
Total blocked 15,000+

These aren't hypothetical. The moment anything is exposed to the internet, the scanners find it within minutes. Watching this in real time made the threat model stuff from my coursework actually click.

Docker + Monitoring Stack

Everything beyond the base OS runs in Docker containers — isolated, easy to manage, easy to update. Portainer gives me a web UI so I'm not SSH-ing in every time I want to restart something. Watchtower runs as a cron job at 3am and pulls updated images automatically. Nginx Proxy Manager sits on port 80 and routes traffic to friendly local domains like grafana.lan instead of raw IP and port combos.

For monitoring I'm running Netdata (real time, 266 metrics, updates every second), Prometheus (scrapes Netdata every 15 seconds so I have historical data), and Grafana (connects to Prometheus and visualizes everything). Uptime Kuma pings all my services every 60 seconds and fires a push notification to my phone the moment anything goes down.

What I Actually Took Away From This

DNS is way more important than it sounds in textbooks. When you control DNS for your whole network you can actually see everything happening on it — malware, ad trackers, sketchy domains, all of it announces itself through DNS before it can do anything. Pi-hole made that visible.

The CrowdSec numbers above are what stuck with me most. I didn't have to do anything to accumulate 15,000 blocked IPs. They were already in the community database from attacks on other people. Defense in depth isn't just a framework concept — I can see each layer doing something different in the logs right now.

The Pi cost under $100. The stack it runs is the same category of tooling that security teams use to protect real infrastructure. That gap between "reading about it" and "running it" is where stuff actually starts to make sense.
